# Concept — 概念模型

## 来龙去脉

这个目录源于一场关于"编程隐式契约"的苏格拉底式对话。

我们逐层构建了八层纵深防御来消灭隐式契约：Rust全栈消除跨语言、隔离仓收口非类型世界、LLM调用序列化/反序列化、事务原语、符号化分析、单线程无共享……前六层都有工具守护——类型系统、隔离仓、事务引擎、符号分析器。

但到了第七层，我们发现了工具的边界：

> 即使所有数据结构都用了强类型的struct，即使有了事务原语保证原子性，
> **事务的边界本身仍然是一个契约**——"哪些操作属于同一个事务"这个知识，
> 在代码演化过程中可能丢失。新来的开发者（或新的AI推理轮次）不知道
> 已有的事务边界在哪里。编译通过，类型正确，但业务语义被破坏了。

解法：**用户心智模型形式化，设计者守关。**

设计者（人类）与agent共创概念模型，在概念设计层面定义事务数量和行为模式。
此形式化直接纳入系统实现的类型保障——代码想声明事务，必须从形式化概念里拿身份。
拿不到身份就编译不过（或CI防御拦截）。只要逼着代码去拿身份，它就会拿到
设计者review过的事务设计，就不会和已有操作脱钩。

这同时解决了三个问题：
1. **事务边界的演化问题**：事务身份是封闭集合，新操作必须归入已有事务或新增事务（需设计者审核）
2. **用户心智模型的落地问题**：概念定义不只是文档，它直接成为系统的类型定义
3. **"谁来守最后一道关"**：前六层工具守，最后一层设计者守

## 当前状态

空的。

概念模型是数据结构之上很高层的东西。目前我们的探讨只到了transaction原语这一步，
还没有成熟到可以落地的程度。需要慢慢摸索。

数据结构（struct）本身已经有完美的类型保护，不需要概念模型的光环。
概念模型要解决的是编译器管不了的业务语义——事务边界、操作归属、行为模式。
